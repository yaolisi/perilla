# Contributing

感谢你对 perilla 的关注。

本项目欢迎 issue、文档修正、测试补充、Bug 修复与功能改进。在提交贡献前，请先阅读本文件与根目录的 `AGENTS.md`。

## 贡献方式

你可以通过以下方式参与：

- 提交 Bug 报告
- 提交功能建议
- 改进文档与示例
- 修复缺陷
- 补充测试与可观测性
- 改进模型接入、Agent、Workflow、Image Generation 等能力

## 开始之前

请先确认：

- 你理解本项目的基本架构：`Web UI` 只是控制台，`FastAPI` 推理网关是统一中枢
- 所有模型调用都必须经过后端网关，不允许前端直连模型或工具
- 新能力优先插件化，避免与核心网关强耦合
- 涉及数据库或向量检索时，必须使用现有 `ORM` 与 `VectorSearchProvider` 抽象

详细约束见：

- `AGENTS.md`
- `docs/architecture/ARCHITECTURE.md`
- `docs/architecture/AGENT_ARCHITECTURE.md`

## 开发环境

建议使用项目当前默认环境：

- Python 3.11+
- Conda
- Node.js 18+
- npm 或 pnpm

后端运行约定：

- 后端相关命令应在项目的 conda 环境中执行

## 基本流程

1. Fork 仓库并创建分支
2. 在本地完成开发与自测
3. 保持改动聚焦，避免无关重构
4. 提交 Pull Request，并说明变更动机、实现方式、影响范围与测试结果

建议分支命名：

- `feature/...`
- `fix/...`
- `docs/...`
- `refactor/...`

## 提交内容要求

### 1. 保持架构边界清晰

请遵循以下原则：

- 不在前端直接接模型、工具或运行时
- 不绕过推理网关
- 不把业务逻辑硬编码到通用 Planner、Runtime、Registry 中
- 新增能力优先做成 Tool / Skill / Plugin，而不是直接塞进核心流程

### 2. 控制改动范围

请尽量做到：

- 一个 PR 聚焦一个主题
- 不顺手修改无关模块
- 不混入大规模格式化噪音
- 不在未明确要求时修改文档、README、架构描述

### 3. 保持可观测性

涉及以下变更时，请补充必要的日志、错误处理或状态信息：

- 推理链路
- Agent / Skill / Tool 执行链路
- Workflow 执行
- 图片生成任务
- 模型加载 / 卸载 / 路由 / 队列

### 4. 保持兼容性

如果改动影响以下能力，请在 PR 描述中明确说明：

- 聊天主链路
- VLM / 多模态
- Agent 执行模式
- Workflow Control Plane
- Image Generation Control Plane
- 模型管理与 `model.json`
- 系统设置与运行时治理

## 代码风格

### Python

- 保持类型标注与清晰边界
- 优先复用现有抽象，不重复造轮子
- 避免在核心路径注入隐式行为
- 错误处理要明确，不吞异常

### 前端

- 保持现有 Vue 3 + TypeScript 风格
- 用户可见文案必须接入 i18n
- 新页面优先复用现有布局与交互模式
- 不把调试信息直接暴露给最终用户

## 测试与验证

提交前建议至少完成与你改动直接相关的验证：

- 后端：最小语法检查 / 单测 / 接口自测
- 前端：页面交互回归 / 关键流程验证
- 模型相关：至少验证模型扫描、加载、执行或卸载中的受影响部分

如果无法完成完整验证，请在 PR 中明确说明未验证项与原因。

## Issue 与 PR 建议

### Bug 报告

建议包含：

- 环境信息（macOS / Ubuntu、Python、Node、运行时）
- 使用的模型或后端
- 复现步骤
- 期望行为
- 实际行为
- 关键日志或截图

### Pull Request

建议包含：

- 背景与目标
- 主要改动
- 架构影响
- 测试结果
- 已知限制或后续工作

## 文档贡献

欢迎修正文档、补充示例与澄清说明。

但请注意：

- 文档内容应与当前实现一致
- 架构文档优先强调边界与职责，不堆实现细节
- API 文档应以当前真实行为为准，不做超前承诺

## 许可证

提交到本仓库的代码默认按项目许可证发布：

- Apache License 2.0

提交贡献即表示你同意你的贡献在该许可证下分发。
