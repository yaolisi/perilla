# Workflow Control Plane V3.0 Gap Analysis（仅保留未完成项与隐患）

更新时间：2026-03-17（按最新实现清理已完成项）

## 1. 当前未完成项（按优先级）

### P0（优先处理）

1. Multi-instance 下治理主状态仍是进程内内存  
- 现象：ExecutionManager 的 queue/slot 在本进程内维护，跨 worker 仅靠 DB running 软兜底。  
- 影响：多进程部署时，全局排队顺序、等待估算、并发视图可能不一致。  
- 建议：将 queue/slot 主状态下沉到共享存储（DB/Redis），引入分布式租约/锁。

2. 后台执行仍基于 API 进程内 `asyncio.create_task`  
- 现象：`wait=false` 执行依赖当前 API 进程生命周期。  
- 影响：进程重启/异常会中断在途任务，恢复依赖 reconcile。  
- 建议：引入独立 worker + 持久化任务队列（至少一次投递 + 幂等执行）。

3. 取消语义仍是“协作式中断”，非抢占式  
- 现象：已支持取消检查与状态回填，但底层长耗时推理调用不可强制打断。  
- 影响：极端场景 Stop 到真正停止仍有尾延迟。  
- 建议：模型/工具适配层补 provider-level cancellation 协议。

### P2（可后置）

1. SQLite 在高并发写场景扩展性有限  
- 现象：取消、对账、节点写回并发时仍会遇到锁竞争。  
- 影响：高峰期延迟与失败率抖动。  
- 建议：中期迁移 PostgreSQL，保留 SQLite 作为单机开发模式。

2. 状态刷新仍以轮询为主（事件化未完成）  
- 现象：运行页与详情页都以轮询为主。  
- 影响：实时性与资源消耗存在折中。  
- 建议：补 SSE/WebSocket 推送节点状态，轮询作为降级路径。

3. 观测指标仍可产品化  
- 现象：有日志与基础状态，但节点级 SLA 指标未系统化展示。  
- 影响：性能调优与故障定位效率受限。  
- 建议：补齐节点维度指标（排队时长、执行时长、重试、token、取消原因）。

## 2. 已识别隐患（持续跟踪）

1. 双状态源长期并存风险（`workflow_executions` vs `graph_instances`）  
- 虽已做 read 路径收敛与 reconcile，仍需持续监控状态漂移。

2. 终态对账（reconcile）在高频轮询下可能触发写放大  
- 建议维持最小间隔与幂等更新，避免重复写回放大锁竞争。

3. 长耗时 Agent 节点可观测断层  
- 需强化“最后活动时间/最后事件时间”展示，避免被误判为卡死。
